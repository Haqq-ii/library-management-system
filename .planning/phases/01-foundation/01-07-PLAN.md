<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → member Server Action | Untrusted caller role + member input |
| action → Better Auth | Credential creation surface |
| action → Prisma | Mass-assignment + hard-delete surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Elevation of Privilege | member actions | mitigate | `requireRole("LIBRARIAN")` first line of every action (T-01); unit test asserts MEMBER rejection. Not middleware |
| T-07-02 | Tampering | mass assignment | mitigate | `MemberSchema.safeParse` with explicit fields; role forced to `"MEMBER"` server-side — caller cannot self-assign `"LIBRARIAN"` (T-02) |
| T-07-03 | Spoofing | weak/raw password | mitigate | User created via Better Auth API (Argon2id); no raw password persisted; min(8) enforced by Zod |
| T-07-04 | Tampering | hard delete losing history | mitigate | Soft delete on `User.deletedAt`; `prisma.user.delete(`/`prisma.member.delete(` banned (INFRA-04, D-14) |
| T-07-05 | Information Disclosure | duplicate-email probing | accept | `EMAIL_EXISTS` is returned only to authenticated LIBRARIANs registering members (not a public signup), so enumeration risk is limited to trusted staff |
</threat_model>
