<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Server Action | Untrusted caller; role must be verified server-side |
| browser cookie → session | Session token presented for every request |
| seed input → DB | Seed writes credentialed users |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | `require-role.ts` | mitigate | `requireRole()` calls `auth.api.getSession()` server-side and throws FORBIDDEN on role mismatch; this is the CVE-2025-29927 defense-in-depth boundary (NOT middleware). Unit test asserts both throw paths |
| T-02-02 | Spoofing | session fixation | mitigate | Better Auth rotates the session token on login; no custom session ID logic in `auth.ts` (T-04 from security_requirements) |
| T-02-03 | Spoofing | open redirect post-login | mitigate | `trustedOrigins: [BETTER_AUTH_URL]` in `auth.ts` constrains callback origins (T-05) |
| T-02-04 | Information Disclosure | seeded passwords | accept | Dev-only seed value `Password123!` in `.env`-gated dev DB; documented as non-production. Production users are created through the real auth flow, never the seed |
| T-02-05 | Tampering | password hashing | mitigate | Seed creates users via Better Auth API (Argon2id internal); no raw password rows, no custom hash |
</threat_model>
