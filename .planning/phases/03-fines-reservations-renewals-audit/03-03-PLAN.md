    New file. Exports: ActionResult type, cancelReservation function.

    "use server" at top. Imports: requireRole, prisma, revalidatePath.

    cancelReservation(reservationId: string): Promise<ActionResult<void>>:
    1. Auth: `let session; try { session = await requireRole("MEMBER"); } catch (err) { return { success: false, error: ... }; }` — captures session for ownership check.
    2. prisma.$transaction(async (tx) => {
       a. tx.reservation.findUnique({ where: { id: reservationId }, include: { member: { include: { user: true } }, book: true } })
       b. If !reservation → throw new Error("NOT_FOUND")
       c. If reservation.member.userId !== session.user.id → throw new Error("FORBIDDEN")
       d. If reservation.status !== "PENDING" → throw new Error("NOT_CANCELLABLE")
       e. tx.reservation.update({ where: { id: reservationId }, data: { status: "CANCELLED" } })
    })
    3. Catch block: handle "NOT_FOUND", "FORBIDDEN", "NOT_CANCELLABLE" with specific error strings; log + return "DB_ERROR" for others.
    4. revalidatePath("/my-reservations") on success.

    Note: Queue position renumbering is NOT required — positions are ordinal display values, not tightly packed; a gap after cancellation is acceptable for v1.

    1. Create (or import from) `src/lib/constants.ts` if it does not exist. The file exports `export const PICKUP_WINDOW_HOURS = 48;`.
    2. Inside reserveBook, after the `if (existing) return { success: false, error: "ALREADY_RESERVED" };` check but before the `prisma.reservation.create`, add lazy expiry logic:

       b. For each expired READY reservation: update status to "CANCELLED".
       c. After expiring, find the AVAILABLE copy that was RESERVED for this expired reservation and find the next PENDING reservation (by queuePosition asc, requestedAt asc).
       d. If a next PENDING reservation exists AND there is a RESERVED copy available: advance that reservation to READY (status = "READY", notifiedAt = now()), set that copy status to "RESERVED" (it already is — just set notifiedAt on reservation).

    Important: The existing reserveBook function does NOT use a prisma.$transaction. Wrap the entire reservation creation block (lazy expiry + create PENDING) in a prisma.$transaction to ensure atomicity. The transaction must include: expire READY reservations → advance next PENDING if applicable → create new PENDING reservation with correct queuePosition.

    The existing queuePosition logic (last?.queuePosition + 1) must remain correct after the transaction wrapper.

    Import PICKUP_WINDOW_HOURS from "@/lib/constants" at the top of catalog/actions.ts.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Spoofing | cancelReservation | mitigate | requireRole("MEMBER") verifies session server-side; LIBRARIAN sessions are rejected |
| T-03-03-02 | Elevation of Privilege | cancelReservation — IDOR | mitigate | Ownership check: reservation.member.userId !== session.user.id → FORBIDDEN; prevents member A from cancelling member B's reservation |
| T-03-03-03 | Tampering | reserveBook lazy expiry | mitigate | Lazy expiry runs inside prisma.$transaction — expiry + PENDING advance + new reservation are atomic; no partial state possible |
| T-03-03-04 | Information Disclosure | /my-reservations | mitigate | Page filters by userId: session.user.id — members see only their own reservations |
| T-03-03-05 | Tampering | queuePosition on cancelReservation | accept | Gaps in queue position after cancellation are acceptable for v1 (noted in action); no integrity risk |
